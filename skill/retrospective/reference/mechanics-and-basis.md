# Механика и доказательная база (reference к retrospective)

Открывать, когда нужно: (а) точно выбрать точку прохождения под урок; (б) свериться с механикой Claude Code (события хуков, лимиты памяти); (в) обосновать подход первоисточниками. Оперативный конвейер — в SKILL.md.

## 1. Ранжированная карта точек прохождения (по убыванию надёжности)

Надёжность = насколько НЕМИНУЕМА точка и детерминированно ли её исполняет harness, а не «решит ли модель». Правило: чем жёстче требуется соблюдение, тем выше по таблице обязан лежать урок. Enforcement («всегда/никогда», блок «готово», блок опасного) — ТОЛЬКО ранг 1.

| Ранг | Куда вшить урок | Надёжность | Природа | Тип урока |
|---|---|---|---|---|
| 1 | **Хук** (PreToolUse / PostToolUse / UserPromptSubmit / SessionStart / Stop / SubagentStop / PreCompact) | Наивысшая — детерминированно исполняет harness | Принуждение | «Всегда/никогда», гейт «готово», блок опасного действия, инъекция правила каждый ход/сессию |
| 2 | Системный промпт / директивы harness | Очень высокая | Принуждение (обычно не под правкой) | Базовые рамки поведения |
| 3 | **CLAUDE.md** (проект/пользователь, авто-грузится целиком каждую сессию) | Высокая по загрузке, МЯГКАЯ по исполнению (advisory) | Совет всегда в контексте | Стабильные конвенции, факты, короткие «делай так», указатели |
| 4 | **Шаблон /goal, slash-команда, DoD-чеклист** внутри скилла | Высокая, ЕСЛИ этот вход реально используется | Процедура | Уроки, привязанные к конкретному запуску/сдаче |
| 5 | **Скилл**, авто-триггер по полю description | Средняя — зависит от совпадения description с задачей | Метод по требованию | Многошаговый метод «как делать X правильно» |
| 6 | **Критик-субагент** (в связке с гейтом ранга 1/4) | Средняя сама по себе, высокая в связке | Проверка в чистом контексте | «Это надо враждебно перепроверять перед готово» |
| 7 | **MEMORY.md индекс** (авто-грузится: первые ~200 строк / 25KB) | Ниже средней | Пассив | Навигатор к деталям |
| 8 | **Детальный memory-файл** (topic-файл, НЕ авто-грузится, читается on-demand) | Низкая (~50%, для сменяющихся сессий почти мёртвая) | Пассив | Архив/справка, на которую ссылается что-то вышестоящее |

Мнемоника: enforcement → хук; метод → скилл; факт/конвенция → CLAUDE.md; сценарий запуска/сдачи → /goal или DoD; враждебная проверка → критик-субагент; «просто на память» → почти не сработает.

## 2. Механика Claude Code (проверено по офиц. докам, актуальный домен code.claude.com, июль 2026)

**Хуки** (`~/.claude/settings.json` → блок `hooks`). Их запускает harness, НЕ модель — единственный по-настоящему принудительный слой.
- События (ядро, имена точные): SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, Stop, SubagentStop, PreCompact, Notification, SessionEnd. (Есть расширенный набор — PermissionRequest, PostToolUseFailure, SubagentStart, FileChanged и др. — для «урок как гарантия» второстепенны.)
- Коды выхода: exit 0 — ок (stdout как JSON; но для UserPromptSubmit и SessionStart сырой stdout идёт в контекст даже без JSON — можно инжектить правило простым echo); exit 2 — блок, stderr уходит модели как ошибка (блокируют: PreToolUse, UserPromptSubmit, Stop, SubagentStop, PreCompact).
- JSON-выход тонкого контроля: `decision:"block"`+`reason`; `hookSpecificOutput.additionalContext` (инъекция контекста); PreToolUse `permissionDecision: allow|deny|ask` и `updatedInput` (переписать аргументы до запуска); PostToolUse `updatedToolOutput`.
- Куда какое событие: правило в каждый ход → UserPromptSubmit; контекст на старте → SessionStart; блок опасного до факта → PreToolUse; блок ложного «готово» → Stop; проверка после действия (линт/тест) → PostToolUse.
- Наши живые примеры «урок → хук»: cto_directive (UserPromptSubmit), require_recommendation (PreToolUse/AskUserQuestion), analytics_canon_gate + guard_no_delete (PreToolUse), proof_gate + capability_claim_gate (Stop).

**Скиллы** (`~/.claude/skills/<name>/SKILL.md`). Progressive disclosure, 3 уровня: (1) `name`+`description` всегда в контексте (десятки токенов); (2) тело SKILL.md грузится, когда скилл сработал; (3) вложенные reference/скрипты — по требованию. Триггеринг: модель авто-подгружает по совпадению запроса с `description` — поэтому description с конкретными «use when…» критичен; абстрактное описание = скилл не выстрелит. Тяжёлое выносить в reference-файлы (как здесь).

**CLAUDE.md / память.** Оба грузятся в начало КАЖДОЙ сессии, НО «Claude treats them as context, not enforced configuration» — гарантирована загрузка, не исполнение. Дисциплина: цель < 200 строк («longer files reduce adherence»); импорты организуют, но НЕ уменьшают контекст. Auto-memory: `~/.claude/projects/<project>/memory/` → грузятся первые ~200 строк / 25KB `MEMORY.md`; topic-файлы при старте НЕ грузятся. Критично: запись ниже 200-й строки MEMORY.md или в topic-файле при старте не видна — «мёртвая», пока что-то не заставит её прочитать. Есть скилл consolidate-memory («Auto Dream») — между сессиями чистит устаревшее, мёржит дубли, разрешает противоречия в источнике, держит лимит.

**Субагенты** (`~/.claude/agents/` или `.claude/agents/`). Свой изолированный контекст: не видит историю/прочитанное — возвращает только сводку/вердикт. Поля frontmatter: `tools`, `model`, `skills` (преднагрузить навыки), `memory` (user/project/local — межсессионное обучение субагента). Критик-субагент видит только дифф+критерии → не предвзят к коду, который сам не писал. Оговорка (из доков): «reviewer prompted to find gaps will usually report some» — критику указывать флагать только то, что бьёт по корректности/требованиям, иначе over-engineering.

## 3. Четыре провала петли самообучения (что убивает «поумнение»)

Из первоисточников (Reflexion, MemGPT, A-MEM, CBR-память, «faulty memory»):
1. **Рефлексия не меняет поведение.** Урок написан прозой, действие осталось прежним. Ядро провала Reflexion/Self-Refine. Лечение: артефакт-механизм + верификация смены поведения диффом (тест, что падал, теперь зелёный).
2. **Провал извлечения (retrieval failure).** Артефакт есть, но не загружен/не сработал: слабый description скилла, запись вне авто-загрузки, «goldfish problem» — знал, но не вспомнил. Лечение: поднять по рангу (хук/авто-загрузка) или отточить description.
3. **Отравление и дрейф памяти (memory poisoning).** Ошибочная/устаревшая запись извлекается и портит поведение; при непрерывном само-переписывании агент может стать ХУЖЕ, чем без памяти. Само-подкрепляющаяся ошибка: ложное правило («X всегда падает») закрывает путь и не даёт собрать опровергающие данные. Лечение: гейт-Revise перед записью (правило, не лаки-случай), провенанс, периодическая прополка (consolidate-memory).
4. **Галлюцинация «фикса».** «Написал сервис/скилл» без проверки, что поток данных реально течёт (наш кейс training_log в несуществующий каталог, ноль записей). Лечение: proof_gate — «готово» только когда COUNT>0 / файл есть / поток переживает диск.

## 4. Доказательная база (принципы петли + первоисточники)

Петля самообучения — синтез из проверенных работ. Ядро каждого механизма:
- **Reflexion** (2303.11366): вербальный урок из ПРОВЕРЕННОГО исхода (тесты/среда) → эпизодическая память → меняет следующую попытку. Провал без надёжного оценщика.
- **Self-Refine** (2303.17651): generate→конкретная само-критика→refine. Потолок = качество самокритики; буксует на строгой логике.
- **Voyager skill-library** (2305.16291): успех → исполняемый навык-код, индекс по эмбеддингу описания, извлечение top-k, композиция навыков. Сохраняется ТОЛЬКО прошедший self-verification. Наш аналог: extract-skills-from-wins.
- **Generative Agents** (2304.03442): извлечение памяти = recency + importance + relevance; reflection поднимает сырьё до правил. Абляции: связка важнее любого одиночного фактора.
- **ADAS / Darwin-Gödel Machine** (2408.08435 / DGM): агент правит свой код, КАЖДОЕ изменение валидируется эмпирически на бенчмарке; архив вариантов против локального оптимума.
- **MemGPT/Letta** (2310.08560) + **A-MEM** (2502.12110): раздели рабочую (горячую, в контексте) и долговременную (холодную, внешнюю) память с явной политикой read/write/evict; связная сеть заметок вместо плоского лога.
- **CBR 4R** (2504.06943) + **ExpeL** (2308.10144): Retrieve→Reuse→Revise→Retain; Revise-шаг отсекает мусор перед сохранением; дистилляция опыта в правила-инсайты.
- **Lean: SDCA→PDCA / hansei / yokoten** (Ohno, Imai «Gemba Kaizen»): «без стандарта нет улучшения»; PDCA поднимает планку → SDCA ФИКСИРУЕТ новый стандарт (удержать выигрыш) → следующий цикл; hansei — плановая честная рефлексия даже при успехе, обязана родить контрмеру-механизм; yokoten — тираж проверенного улучшения горизонтально.
- **Ретро/постмортем-инженерия**: blameless (Google SRE) — по роли, не по имени; just-culture (Reason/Dekker) — различать честную ошибку / рискованное / безрассудство, разбирать и near-miss; иерархия контролей (NIOSH/OSHA); poka-yoke control>warning (Shingo); A3/PDCA «контрмера = гипотеза, не решение»; критика «5 почему» (Dekker «нет единственного корня») → несколько условий; провал action items ~35-50% доводимости без владельца/трекинга → один владелец, лимит 1-3, стоячая повестка.
