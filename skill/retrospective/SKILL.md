# Ретро/постмортем → петля самообучения

Цель скилла — не «написать разбор», а сделать так, чтобы **каждая сессия оставляла систему умнее**: опыт (провал ИЛИ выигрыш) превращается в артефакт, который неминуемо сработает в следующих сессиях. Это петля из работ по самообучающимся агентам (Voyager, Reflexion, CBR) + инженерии постмортемов + lean (SDCA→PDCA/hansei). Детали и источники — reference/mechanics-and-basis.md.

## Главный закон (доказанная проблема)
Урок надёжен ровно настолько, насколько **неминуема точка прохождения**, где он лежит, и **доказано, что он сменил поведение**. Проза (память, заметка, абзац отчёта) — нижняя ступень: базовая надёжность ~50%, а для сменяющихся LLM-сессий ещё ниже — сессия физически проходит только через системный промпт, CLAUDE.md, шаблон /goal, скиллы, гейты/хуки и авто-загружаемый MEMORY-индекс (первые ~200 строк). Урок вне этих точек не сработает. Доказательство: `domain-census-before-depth` БЫЛ записан в память и всё равно не предотвратил потерю обучающего корпуса на Grantum. **Фиксировать урок прозой = почти гарантировать рецидив.**

## Петля самообучения (спина скилла — 8 шагов)
Каждый пойманный опыт гони по кольцу:
1. **Захват (Capture).** Зафиксируй из ПРОВЕРЕННОГО исхода, не из самомнения. Провал → постмортем; выигрыш → «что стандартизируем» (hansei честен и при успехе). Заземляй в код/БД/файлы, проверяй руками, не по памяти.
2. **Дистилляция (Distill).** Подними прозу до артефакта: правило → чек/хук; приём → скилл/шаблон; урок сдачи → пункт DoD. Не «запомнить, что X».
3. **Гейт-Revise (перед записью).** Прежде чем вшивать: воспроизводимо? это ПРАВИЛО или лаки-случай? не противоречит существующему? Отсекает отравление памяти и «удачу как правило».
4. **Маршрутизация (Route).** Отправь артефакт в правильную точку прохождения по иерархии контролей (ниже) — максимально высоко.
5. **Индексация для извлечения (Retrieve).** Артефакт бесполезен, если не сработает в нужный момент: у скилла — точный `description` («use when…»); у правила — авто-загрузка/хук, а не topic-файл вне контекста. Провал извлечения = мёртвый урок.
6. **Верификация смены поведения (Verify).** Урок НЕ закрыт, пока не показан дифф поведения: гейт ловит регресс / тест, что падал, теперь зелёный / повтор задачи проходит без старой ошибки. «Написал» ≠ «изменил».
7. **Стандартизация (SDCA — удержать выигрыш).** Новый приём → новый «пол»: зафиксируй как стандарт, чтобы не сползло назад. Только со стабильной базы — следующее улучшение.
8. **Check + тираж (yokoten).** Назначь дату/триггер повторной проверки «сработало?» (обычно — первый пункт следующего ретро). Проверенный переносимый урок — разнеси во все релевантные проекты/скиллы.

## Иерархия контролей (куда вшивать — по убыванию надёжности)
1. **Устранение** — переконструировать так, чтобы ошибка стала невозможна (схема БД NOT NULL/append-only; единственный способ вставки).
2. **Инженерный барьер (poka-yoke)** — конструкция не даёт сделать неправильно (CHECK-constraint, обязательный аргумент, единый chokepoint вместо 20 точек). Control сильнее warning.
3. **Автоматическое обнаружение** — проверка останавливает процесс при нарушении (тест, линтер, CI-гейт, смок «файл появился», хук).
4. **Административная мера** — пункт definition-of-done скилла/спеки, обязательный шаг шаблона /goal, стоячая повестка.
5. **Напоминание** — заметка в памяти, абзац в отчёте. САМЫЙ слабый; для LLM-сессий почти мёртвый. Допустим ТОЛЬКО дублем к механизму ступеней 1–4.

Правило: для каждого урока спроси «на какую ступень поднять?» и подними максимально.

## Карта маршрутизации: тип урока → точка прохождения (кратко)
- «Всегда/никогда», блок «готово», блок опасного, инъекция правила каждый ход → **ХУК** (наивысшая надёжность; PreToolUse/Stop/UserPromptSubmit/SessionStart/PostToolUse). Плюс критик-субагент как исполнитель проверки.
- «Метод/процедура, как делать X правильно» → **СКИЛЛ** с отточенным `description`; если метод обязателен на входе/сдаче — продублируй вызов в шаблон /goal или DoD.
- «Стабильный факт/конвенция/указатель» → **CLAUDE.md** коротко; детали в файл, в CLAUDE.md/MEMORY.md — ссылка.
- «Привязан к конкретному запуску/сдаче» → **шаблон /goal или slash-команда**.
- «Надо враждебно перепроверять перед готово» → **критик-субагент** в связке с гейтом.
Полная ранжированная таблица + механика (события хуков, лимиты памяти) — reference/mechanics-and-basis.md §1–2.

## Четыре провала петли (проверяй, что не попал в них)
1. Рефлексия не сменила поведение (проза без механизма) → артефакт + дифф-верификация.
2. Провал извлечения (артефакт есть, но не грузится/не триггерится) → выше по рангу или точнее `description`.
3. Отравление/дрейф памяти (устаревшее/ошибочное извлекается и вредит) → гейт-Revise + прополка (consolidate-memory).
4. Галлюцинация «фикса» («написал сервис» без текущего потока данных) → proof_gate: COUNT>0/файл есть/поток переживает диск.
Подробно — reference §3.

## Правила ретро-ритуала
- **Blameless + just-culture.** Причина ВСЕГДА в среде (промпт/спека/нет гейта/рамка задачи), не «сессия была невнимательна» — у безпамятной сессии винить некого, чини среду. Но различай: честная ошибка → чини систему; рискованный срез угла → убери стимул; безрассудство → ответственность. Разбирай и near-miss (пронесло), не только реализовавшиеся провалы. «Надо было внимательнее» = сигнал, что механизм не найден.
- **Условия, а не «корень» (how, not why).** Собери **2–3 contributing conditions** («КАК это стало возможным»), не «5 почему» (ложная единственность, соскальзывает в «кто»). Дополни вопросом «почему это выглядело разумным тогда?».
- **Стоячая повестка.** Каждое ретро НАЧИНАЙ с ревизии прошлых action items (сделано / почему застряло) — это поднимает их исполнение на +15–25 п.п. Без этого петля не замыкается.
- **Лимит 1–3 экшена.** Больше → перегрузка и падение доводимости. Остальное осознанно отложи, не тащи всё.
- **Census-продолжение.** Найдя упущение класса X — прогони перепись ОТ ДОЛГОСРОЧНОЙ ЦЕЛИ назад: «где ещё этот класс?». Одно упущение рамки почти всегда = серия. (Связка с [[domain-census-before-depth]], скилл deep-problem-solving.)

## Формат action item (живой, не мёртвый)
Каждый пункт ОБЯЗАН иметь:
- **Глагол + конкретика + границы:** ясно, когда «сделано». Не «улучшить сбор данных» → «добавить append-врезку (input_sha256, teacher_output, model, prompt_rev) в ai_service._call».
- **Владелец-МЕХАНИЗМ, не память:** файл/гейт/хук/схема/шаблон, который исполнит без чьей-либо воли. «Помнить, что X» — запрещено. Валидно: «добавить X в гейт Y» / «CHECK-constraint в миграции» / «пункт DoD в шаблоне» / «хук в settings.json».
- **Ступень контроля (1–5)** + план подъёма выше прозы. Стоит 4–5 — в строке причина, почему не подняли.
- **Контрмера — это ГИПОТЕЗА, не «решение»** (A3/PDCA): «ожидаем, что устранит X; проверим к Y по метрике Z».
- **Дата/триггер Check** + один владелец. Приоритет: минимум один P0/P1.

## Стандартная форма документа (фиксированный скелет [1]→[2]→[3])
Всегда: **[1] шапка-сводка → [2] детали → [3] подвал-план**. Шапка и подвал компактные/табличные (читаются за 15 сек), детали между ними можно пропустить.

**[1] ШАПКА (BLUF, ≤8 строк):**
```
## Итог
**Ключевой вывод:** <1 фраза>
**Сработало (стандартизируем):** <1 строка>
**Главный провал:** <что и почему — условие, не «кто»>
**Счёт:** N уроков · M экшенов · ✅K / ⏳P / ❌L
**Ревизия прошлых экшенов:** <из прошлого ретро: сделано/застряло>
```

**[2] ДЕТАЛИ:** инцидент(ы) и что невосстановимо; 2–3 условия; census-продолжение; «что сработало». Всё заземлено в код/БД.

**[3] ПОДВАЛ — ЭКШЕН-ПЛАН (главная таблица, ≤7 строк):**
```
## Что дальше
| # | Экшен (глагол+конкретика, ясно «когда сделано») | Ступень | Точка прохождения (не память!) | Приоритет | Статус | Verify+Check |
|---|---|---|---|---|---|---|
| 1 | … | 1–5 | миграция/гейт/хук/скилл/DoD | P0 | ✅/⏳/❌ | дифф поведения · дата · кто |
```
Под таблицей — **СЛЕДУЮЩИЙ ШАГ одной строкой.** Сортировка: P0 сверху; среди равных — ❌ выше ✅. Статус ЧЕСТНЫЙ: ✅ только если механизм существует И поток течёт (COUNT>0/файл есть); «написал спеку/код» без потока = ⏳/❌.

## Выход (артефакты, не проза)
1. Документ по скелету [1]→[2]→[3].
2. **Реальные правки исполняемых артефактов в той же сессии:** хук/гейт, тест/смок, схема БД, пункт DoD, шаблон /goal, новый или обновлённый скилл. Это и есть «закрытие».
3. Заметки в память — ТОЛЬКО как индекс/дубль к механизму (правило [[extract-skills-from-wins]]), со ссылкой на механизм-носитель, в первых 200 строках MEMORY.md.
4. Переиспользуемый за пределами проекта урок — заскиллить (yokoten).

## Анти-паттерны
- «Запишу в память/отчёт» как единственное действие — рецидив почти гарантирован.
- Один «корень» вместо нескольких условий.
- Контрмера-напоминание там, где возможен барьер/тест.
- «Фикс = написал сервис» без проверки, что поток данных реально течёт и переживает диск.
- Action item без владельца-механизма, без ступени, без дифф-верификации и даты Check.
- Урок закодировал лаки-случай как правило (пропущен гейт-Revise) → само-подкрепляющаяся ошибка.
- Разбор остановлен на «надо внимательнее» (механизм не найден).
- Ретро без ревизии прошлых экшенов; список из 7+ пунктов.

## Быстрый маршрут (мало времени)
1. Что упущено/выиграно, что невосстановимо (факт, не память).
2. 2–3 условия «как стало возможным» (не «кто»).
3. На каждое — контрмера-механизм + ступень + точка прохождения (хук/скилл/гейт/DoD/схема).
4. Где ещё этот класс? (census от цели).
5. Внеси ≥1 механизм СЕЙЧАС, проверь дифф поведения, назначь Check первым пунктом следующего ретро. Лимит 1–3 экшена.
