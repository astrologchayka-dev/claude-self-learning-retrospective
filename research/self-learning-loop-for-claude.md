# Петля самообучения для агентов Claude Code: ресёрч и синтез

Как сделать так, чтобы кодинг-агент (Claude Code и подобные) **становился умнее с каждой сессией**, а не начинал каждый раз с нуля. Сведение доказательной базы из четырёх областей в один рабочий метод. В конце — полный список источников.

> Практический вывод одной фразой: **урок надёжен ровно настолько, насколько неминуема точка прохождения, где он лежит, и доказано, что он сменил поведение.** Заметка в памяти — низшая ступень (~50% надёжности; для сменяющихся LLM-сессий почти мёртвая).

---

## 0. Постановка проблемы

LLM-сессия без памяти физически проходит только через фиксированный набор точек: системный промпт, авто-загружаемый CLAUDE.md, шаблоны запуска, скиллы (авто-триггер по описанию), хуки на событиях и авто-загружаемый индекс памяти (первые ~200 строк). Урок, лежащий вне этих точек, в следующей сессии просто не будет прочитан. Значит «зафиксировать опыт прозой» = почти гарантировать рецидив. Задача петли самообучения — поднять каждый урок с уровня прозы до уровня механизма, который сработает неминуемо.

## 1. Что говорит наука о самообучающихся агентах

### Reflexion (Shinn и др., 2023)
«Вербальное подкрепление». Три роли: Actor действует, Evaluator оценивает исход (тесты/награда среды), Self-Reflection превращает провал в текстовый урок «почему не вышло и что менять». Урок пишется в эпизодическую память и подкладывается в следующую попытку. Веса не трогаются — учится только текст.
- **Что делает надёжным:** урок опирается на ПРОВЕРЕННЫЙ исход, а не на самомнение. HumanEval pass@1 ~91% (vs 80% GPT-4); на ALFWorld галлюцинации за 10 попыток упали 32%→3%.
- **Провал:** без надёжного оценщика рефлексия сваливается в правдоподобную пустоту; «урок написан — поведение то же»; на MBPP-Python метод даже ухудшил результат из-за ложноположительных self-тестов.

### Self-Refine (Madaan и др., 2023)
Одна модель без обучения: generate → конкретная само-критика → refine, цикл. Всё внутри одного запроса, без внешней памяти. Средний прирост ~20 п.п. на 7 задачах.
- **Что делает надёжным:** критика должна быть адресной («пункт X нарушает Y»), не «сделай лучше».
- **Провал:** потолок = способность модели УВИДЕТЬ свою ошибку; на строгой математике прирост ≈0. Модель хуже дискриминирует свои ответы, чем генерирует.

### Voyager (Wang и др., 2023) — паттерн «библиотека навыков», ключевой
Агент в Minecraft: (a) авто-курриулум сам ставит всё более сложные цели; (b) **skill library** — успешная траектория оформляется в исполняемый код-функцию, к ней LLM пишет описание, навык индексируется по эмбеддингу описания и позже извлекается по запросу-цели top-k; (c) итеративный промптинг с обратной связью среды + self-verification ПЕРЕД сохранением. Навыки композируются: сложное строится из простого.
- **Что делает надёжным:** артефакт — исполняемый код, а не заметка (работает/не работает); self-verification = гейт перед записью; библиотека компаундится. Против AutoGPT/ReAct/Reflexion: ×3.3 уникальных предметов, единственный дошёл до алмаза, zero-shot перенос в новый мир.
- **Провал:** галлюцинации кода; навык кодирует лаки-случай, а не правило; ошибка извлечения (по описанию достаётся не тот навык); разрастание библиотеки без прополки.

### Generative Agents (Park и др., 2023) — извлечение памяти
Memory stream — хронологический лог наблюдений. Извлечение ранжирует по трём сигналам: **recency** (спад по времени) + **importance** (LLM ставит 1–10) + **relevance** (косинус эмбеддингов). Reflection: при накоплении важности агент синтезирует из сырья абстрактные выводы и кладёт обратно.
- **Что делает надёжным:** три сигнала бьют по разным типам промахов извлечения; рефлексия поднимает уровень абстракции (правила вместо сырья). Абляции: связка важнее любого одиночного фактора.
- **Провал:** нерелевантное извлечение, «приукрашивание» выводов, дороговизна.

### ADAS / Darwin-Gödel Machine (2024–2025) — самоэволюция
Мета-агент программирует новых агентов кодом и копит **архив** вариантов; Darwin-Gödel Machine переписывает собственный код и валидирует КАЖДОЕ изменение **эмпирически** на бенчмарке (SWE-bench/Polyglot), а не «звучит лучше».
- **Что делает надёжным:** эмпирическая валидация каждого изменения; архив = защита от локального оптимума (слабый сейчас вариант — трамплин позже).
- **Провал:** reward hacking, дороговизна, риски самомодификации.

### MemGPT/Letta (2023) + A-MEM (2025) — уровни памяти
Аналогия «LLM как ОС»: окно контекста = оперативка (горячее), внешнее хранилище = диск (холодное), LLM сам управляет пейджингом через функции. Явные `core_memory_append/replace`, `archival_memory_insert/search`. A-MEM добавляет связную сеть заметок (Zettelkasten) вместо плоского лога. MemGPT: Deep Memory Retrieval 32%→92% с GPT-4.
- **Принцип:** раздели рабочую (в контексте, малая, горячая) и долговременную (внешняя, большая, холодная) память с явной политикой read/write/evict.
- **Провал:** галлюцинированные правки памяти, retrieval-but-unused, засорение контекста, агент плохо управляет своей памятью, **memory poisoning**.

### CBR (4R) + ExpeL (2023–2025) — опыт как переиспользуемые кейсы
Цикл **Retrieve → Reuse → Revise → Retain**: храни прошлые эпизоды решения как кейсы, извлекай похожие, адаптируй, ДОРАБОТАЙ (Revise) и сохрани новый. ExpeL дистиллирует опыт множества задач в естественно-языковые инсайты + переиспользует успешные траектории как few-shot.
- **Что делает надёжным:** хранится целый разобранный эпизод + правило; Revise-шаг отсекает мусор перед Retain.
- **Провал:** извлечь не тот кейс; negative transfer (правило из одного домена ломает другой); устаревшие кейсы; **само-подкрепляющаяся ошибка** — ложное правило («API X всегда падает») закрывает путь и не даёт собрать опровергающие данные; при непрерывном само-переписывании памяти агент может стать ХУЖЕ, чем без памяти.

### Lean / Toyota (Ohno, Imai) — как разовое становится стандартом
- «**Без стандарта нет улучшения**» (Тайити Оно): документированный стандарт — предпосылка любого кайдзен.
- **SDCA ↔ PDCA** (Масааки Имаи): PDCA поднимает планку → **SDCA фиксирует новый стандарт** (удержать выигрыш, не сползти назад) → только со стабильной базы следующий цикл. «Сначала стабилизируй, потом улучшай.»
- **Hansei** — плановая честная рефлексия ДАЖЕ при успехе, обязана родить контрмеру-механизм, а не заметку.
- **Yokoten** — горизонтальный тираж проверенного улучшения на всю организацию.
- **Провал при переносе на агента:** урок остался «заметкой», не стал стандартом на неминуемом пути → дрейф назад; не растиражирован → умер в одной сессии.

## 2. Инженерия постмортемов (чтобы разбор реально предотвращал рецидив)

- **Blameless** (Google SRE): по роли, не по имени; каждый действовал на основе той информации, что имел. Фиксированный шаблон, запускаемый по заранее заданным критериям.
- **Action items проваливаются** без владельца и трекинга (доводимость по практическим обзорам ~35–50%). Лечение: **один владелец** на пункт; **лимит 1–3 пункта** (7 → перегрузка); **стоячая повестка** — каждое ретро НАЧИНАЕТСЯ с ревизии прошлых пунктов (+15–25 п.п. к исполнению); пункт улучшения — в тот же поток работы, что и обычные задачи.
- **Иерархия контролей** (safety science): устранение > замена > инженерный контроль > административный > СИЗ/предупреждение. Первые три работают БЕЗ участия человека — потому надёжнее.
- **Poka-yoke** (Shingo): **control** (не даёт браку пройти) сильнее **warning** (просто сигналит).
- **A3 / PDCA:** контрмера — это ГИПОТЕЗА, проверяемая экспериментом («ожидаем X, проверим к Y по метрике Z»), а не финальное «починили».
- **Критика «5 почему»** (Dekker, «нет единственного корня»): сбой рождается из совпадения нескольких нормальных условий. Ищи **2–3 contributing conditions** + вопрос «почему это выглядело разумным тогда?».
- **Just culture** (Reason/Marx/Dekker): различай честную ошибку (чини систему) / рискованное поведение (убери стимул) / безрассудство (ответственность). Разбирай и near-miss, не только реализовавшиеся провалы.

## 3. Механика Claude Code: куда физически класть урок

Проверено по официальным докам (актуальный домен `code.claude.com`, 2026). Надёжность = насколько неминуема точка и детерминированно ли её исполняет harness, а не «решит ли модель».

### Ранжированная карта точек прохождения

| Ранг | Куда | Надёжность | Природа | Тип урока |
|---|---|---|---|---|
| 1 | **Хук** (PreToolUse / PostToolUse / UserPromptSubmit / SessionStart / Stop / SubagentStop / PreCompact) | Наивысшая — исполняет harness | Принуждение | «Всегда/никогда», гейт «готово», блок опасного, инъекция правила каждый ход |
| 2 | Системный промпт / директивы harness | Очень высокая | Принуждение (не под правкой) | Базовые рамки |
| 3 | **CLAUDE.md** (авто-грузится целиком каждую сессию) | Высокая по загрузке, мягкая по исполнению | Совет всегда в контексте | Стабильные конвенции, факты, указатели |
| 4 | **Шаблон запуска / slash-команда / DoD-чеклист** | Высокая, если вход используется | Процедура | Уроки, привязанные к запуску/сдаче |
| 5 | **Скилл**, авто-триггер по `description` | Средняя — зависит от совпадения описания | Метод по требованию | Многошаговый метод «как делать X» |
| 6 | **Критик-субагент** (в связке с гейтом ранга 1/4) | Средняя сама по себе, высокая в связке | Проверка в чистом контексте | «Враждебно перепроверять перед готово» |
| 7 | **MEMORY.md индекс** (авто: первые ~200 строк / 25KB) | Ниже средней | Пассив | Навигатор к деталям |
| 8 | **Topic-файл памяти** (не авто-грузится, on-demand) | Низкая (~50%, почти мёртвая) | Пассив | Архив, на который ссылается вышестоящее |

**Мнемоника:** enforcement → хук; метод → скилл; факт → CLAUDE.md; сценарий → шаблон/DoD; враждебная проверка → критик-субагент; «просто на память» → почти не сработает.

### Ключевые факты механики
- **Хуки** запускает harness, не модель — единственный принудительный слой. UserPromptSubmit и SessionStart принимают сырой stdout прямо в контекст (можно инжектить правило простым echo). exit 2 блокирует (PreToolUse/UserPromptSubmit/Stop/SubagentStop/PreCompact). JSON-выход: `decision:"block"`, `hookSpecificOutput.additionalContext`, PreToolUse `permissionDecision` + `updatedInput`.
- **Скиллы** — progressive disclosure: `name`+`description` всегда в контексте; тело грузится при срабатывании; reference-файлы — по требованию. Триггеринг по совпадению с `description` → конкретные «use when…» критичны.
- **CLAUDE.md / память** — «treated as context, not enforced configuration»: загрузка гарантирована, исполнение нет. Цель < 200 строк («longer files reduce adherence»). Auto-memory: грузятся первые ~200 строк / 25KB `MEMORY.md`; topic-файлы при старте НЕ грузятся → запись ниже 200-й строки «мёртвая». Для enforcement — хук, не память.
- **Субагенты** — изолированный контекст (возвращают сводку/вердикт). Критик видит только дифф+критерии → не предвзят. Оговорка: «reviewer prompted to find gaps will usually report some» — указывай флагать только то, что бьёт по корректности.

## 4. Синтез: петля самообучения (8 шагов)

Каждый пойманный опыт (провал ИЛИ выигрыш) гони по кольцу:

1. **Захват (Capture)** — из ПРОВЕРЕННОГО исхода, не из самомнения (Reflexion, DGM). Заземляй в код/БД/файлы.
2. **Дистилляция (Distill)** — подними прозу до артефакта: правило → чек/хук, приём → скилл/шаблон, урок сдачи → пункт DoD (Voyager code-skill, ExpeL insight, hansei→контрмера).
3. **Гейт-Revise** — перед записью: воспроизводимо? это правило или лаки-случай? не противоречит существующему? (CBR Revise, Voyager self-verification). Отсекает отравление памяти.
4. **Маршрутизация (Route)** — в правильную точку прохождения по иерархии контролей, максимально высоко.
5. **Индексация для извлечения (Retrieve)** — точный `description`/индекс; привязка к неминуемой точке, а не «вспомнит ли модель» (Voyager retrieval, Generative Agents relevance).
6. **Верификация смены поведения (Verify)** — урок закрыт только с диффом: гейт ловит регресс / тест, что падал, теперь зелёный (ключевой провал Reflexion/Self-Refine — «написал ≠ изменил»).
7. **Стандартизация (SDCA)** — зафиксируй как новый «пол», чтобы не сползло (Imai). Только потом следующее улучшение.
8. **Check + тираж (yokoten)** — дата/триггер повторной проверки (первый пункт следующего ретро); проверенный переносимый урок — во все релевантные проекты.

### Десять принципов (сжато)
1. Учись только на проверенном исходе (нужен внешний оценщик).
2. Дистиллируй в исполняемый/проверяемый артефакт, а не в заметку.
3. Гейт перед записью (правило, не удача).
4. Раздели рабочую и долговременную память с явной политикой.
5. Индексируй для извлечения + привяжи к неминуемой точке.
6. Ранжируй память по нескольким сигналам (recency+importance+relevance), поднимай сырьё до правил рефлексией.
7. Верифицируй, что артефакт РЕАЛЬНО сменил поведение (дифф).
8. Сначала удержи выигрыш (SDCA), потом улучшай.
9. Держи архив вариантов против локального оптимума.
10. Тиражируй горизонтально (yokoten) и защищайся от отравления памяти (провенанс, прополка).

### Четыре провала петли (чего избегать)
1. **Рефлексия не сменила поведение** — проза без механизма. → артефакт + дифф-верификация.
2. **Провал извлечения** — артефакт есть, но не грузится/не триггерится. → выше по рангу или точнее `description`.
3. **Отравление/дрейф памяти** — устаревшее/ошибочное извлекается и вредит. → гейт-Revise + прополка.
4. **Галлюцинация «фикса»** — «написал сервис» без текущего потока данных. → пруф-гейт (COUNT>0 / файл есть / поток переживает диск).
