# Экономика контекста Claude Code: как тратить меньше токенов и получать выше качество

Ресёрч 2026-07-01. Два независимых прохода: официальные источники Anthropic (документация + инженерный блог) и полевые практики (замеры heavy-юзеров, GitHub-issues, исследование Chroma «Context Rot»). Плюс применение на живой системе (кейс в конце).

## Главный вывод

Экономия токенов и качество ответов — одна и та же ручка, а не трейд-офф. Деградация точности от длины контекста измерена (Chroma «Context Rot», 18 фронтир-моделей): заметное падение уже с ~50 КБ входа, −7,9% чисто от длины даже без отвлекающего мусора. Всё, что грузится «на каждый ход» — CLAUDE.md, индекс памяти, описания скиллов, текст хуков, схемы MCP — это налог одновременно на деньги и на точность. Базовая стратегия: переносить знание из «всегда в контексте» в «по требованию».

## 1. Что платится на каждом ходу

- CLAUDE.md читается каждый ход каждой сессии: файл на 5000 токенов = 5000 токенов на каждое сообщение. Целевой размер по полю — 300–600 токенов; тест на строку: «уберу — Claude начнёт ошибаться?» Нет → удалить. Замер: сокращение 3847 → 312 токенов без падения качества.
- Импорты @path в CLAUDE.md не экономят — импортированное грузится на старте. Вместо инлайна давать путь + «когда читать».
- Не дублировать в CLAUDE.md то, что enforce'ит инструмент (линтер, хук-гейт) — инструмент и есть источник правды.
- Хук-инжекты на частых событиях (UserPromptSubmit, PreToolUse) — тот же налог. Писать их как факты, не императивные команды: императив триггерит защиту от prompt-injection и текст не применяется. Скрипт хука держать <100 мс.

## 2. Память (auto memory)

- Система грузит только первые 200 строк / 25 КБ файла MEMORY.md — всё сверх молча обрезается. Это тихий баг качества: хвост индекса просто исчезает для модели.
- Правильная форма: индекс = одна строка на факт-файл, контент — в отдельных файлах, читаемых по требованию.
- Устаревшие записи активно вредят (модель следует стухшим фактам) — нужен регулярный consolidate/prune, а лучше — автоматический гейт (см. кейс).

## 3. MCP-серверы

- Классическая загрузка: каждый сервер инжектит все схемы тулов в каждый ход — 10–20 К токенов на сервер; в issues задокументировано «hi» за ~31 К токенов и до 50% окна на тулы.
- Штатный фикс — deferred loading / Tool Search: −85% оверхеда (замер Anthropic), −46,9% в полевом замере. Но список серверов всё равно держать минимальным: сотня отложенных имён — это и токены, и соблазн выбрать не тот инструмент.
- Дизайн сервера: консолидация 4 похожих тулов в 1 с параметром = −60% схем (замер Scott Spence). Меньше продуманных тулов лучше, чем тул на каждый эндпоинт.
- Радикальная полевая позиция: MCP только для stateful/secure-шлюзов; всё stateless (GitHub, Jira, облака) дешевле через CLI в Bash — CLI не платит схемный налог и композируется в скриптах.

## 4. Кэширование

- На подписке Claude Code кэш живёт 1 час (на API-ключе — 5 минут). Паузы в работе почти не штрафуются.
- Кэш ломают: смена модели посреди сессии, включение/выключение MCP-сервера, /compact, обновление Claude Code, resume старой сессии после апдейта. Каждая поломка = полная переплата за весь контекст.
- Правило: модель выбрать в начале сессии и не переключать; plan mode безопасен (инжектится сообщением, не меняет системный промпт).

## 5. Сессии и гигиена контекста

- Новая задача = /clear или новая сессия — консенсус и доков, и поля. История прошлой задачи — мёртвый груз и источник отравления контекста.
- /compact — только на естественной паузе, не в середине работы; авто-compact непрозрачен, поле предпочитает «Document & Clear»: резюме в файл → очистка → продолжение от файла.
- Хендофф-файл писать, пока осталось ≥20% окна — позже качество резюме падает. Структура: цель / сделано / решения / файлы / открытые вопросы / следующий шаг / «отброшено как шум» (провальные гипотезы, которые НЕ должны переехать в новую сессию).
- Если поправил модель >2 раз на одной проблеме — контекст отравлен неудачными попытками; дешевле /clear и новый заход с лучшей постановкой.

## 6. Субагенты

- Субагенты НЕ экономят по умолчанию: замеры дают 4–7× токенов (каждый — свежий контекст, перечитывающий базу).
- Окупаются на шумной работе: разведка кода, прогон тестов/логов, веб-ресёрч — в основную сессию возвращается выжимка на 1–2 К токенов вместо тонны сырья.
- Не окупаются на мелких последовательных шагах, где шаг 2 требует полный вывод шага 1.
- Дешёвая модель (haiku) для простых субагентов — основной источник экономии на делегировании.

## 7. Скиллы

- Описания скиллов живут в бюджете ~1% окна; описание длиннее 1536 символов режется, а редкоиспользуемые скиллы при переполнении молча отключаются целиком. Проверять /doctor; описания держать короткими, редкое — архивировать.
- Раздутый SKILL.md деградирует выполнение так же, как раздутый контекст: тело короткое, детали — в reference-файлах по ссылке (progressive disclosure).
- В CLAUDE.md — только то, что нужно каждой сессии; редкие многошаговые процедуры — в скиллы.

## 8. Кейс: система бюджетов контекста (применение на живой системе)

Найденный на практике баг: MEMORY.md вырос до 26,5 КБ при потолке 25 КБ — хвост индекса уже молча обрезался. Разовое сжатие — не решение (через неделю снова вырастет). Решение по принципу «урок = механизм в неминуемой точке»:

1. Бюджеты-числа: индекс памяти ≤20 КБ (тревога) / 24 КБ (блок); CLAUDE.md ≤12 КБ (тревога); описание скилла ≤1536 симв.
2. Гейт на Stop-хуке (context_budget_gate.py): при превышении блокирует завершение хода с инструкцией «сожми индекс» — ни одна сессия не может молча закончить работу при переполненной памяти. Проверено: при 26,5 КБ блокирует, после сжатия до 16 КБ пропускает.
3. Правило записи в шапке самого MEMORY.md: новая запись = 1 строка, закрытая задача → свёртка в строку-указатель, детали всегда в файле.

Паттерн переносим: любой «ползучий» ресурс (индекс, конфиг, список правил) = число-бюджет + гейт в неминуемой точке + правило записи в самом артефакте.
